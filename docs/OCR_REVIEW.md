# OCR 로직 검토 — 놓친 부분·성능 개선 여지

현재 구현을 기준으로 **누락 가능성**과 **테서랙트 trained data처럼 쉽게 쓸 수 있는 성능 향상 방법**을 정리한 문서다.

> **반영 완료:** `get_ocr_engine()` 에 `use_gpu`, `show_log`, `drop_score`, `rec_char_dict_path` 인자를 추가해 두었음. 필요 시 라우터/설정에서 넘겨서 사용하면 됨.

---

## 1. 현재 로직에서 점검한 부분

| 항목 | 상태 | 비고 |
|------|------|------|
| PDF → 이미지 | ✅ | PyMuPDF (시스템 Poppler 불필요) |
| OCR 엔진 재사용 | ✅ | 싱글톤 `get_engine()`, 첫 요청 시 1회만 로드 |
| 블로킹 회피 | ✅ | `asyncio.to_thread(_run_ocr_sync)` |
| 3GB GPU 대응 | ✅ | max_side_len=1280, use_angle_cls=False, 페이지별 처리 |
| 한글 | ✅ | `lang="korean"` |
| 임시 파일 정리 | ✅ | `finally` 에서 `unlink(missing_ok=True)` |

---

## 2. 놓치기 쉬운 부분

### 2.1 PaddleOCR 초기화 옵션

- **`show_log=False`**  
  터미널 로그를 줄이고 싶으면 생성자에 넣을 수 있음. (이미 앱 전역에서 로거 레벨로 억제 중이라 필수는 아님.)

- **`use_gpu`**  
  GPU 버전 설치 시 기본값이 `True`라 동작에는 문제 없음. 코드에서 명시해 두면 “의도적으로 GPU 사용”이 드러나서 유지보수에 유리함.

- **`drop_score`** (기본 0.5)  
  **인식 신뢰도가 이 값 미만인 결과는 버림.**  
  - 품질을 높이려면 → 값을 올리면 됨 (예: 0.6~0.7).  
  - 오인식이 많으면 → 값을 내리면 됨 (예: 0.3~0.4).  
  현재는 기본값만 쓰고 있어서, 문서 특성에 맞게 **한 번 조정해 보는 것**을 권장.

- **`rec_batch_num`** (기본 30)  
  한 이미지 안에서 인식(rec)할 때 묶는 개수.  
  - 3GB VRAM에서 OOM이 나면 **줄이기** (예: 10).  
  - 페이지당 텍스트가 많지 않으면 현재처럼 두어도 됨.

### 2.2 테서랙트 “trained data”에 해당하는 것 — 커스텀 사전

- **`rec_char_dict_path`**  
  **인식(rec) 모델이 쓰는 문자 사전 파일 경로**를 지정하는 옵션이다.  
  - 테서랙트의 추가 tessdata/사전처럼, **별도 학습 없이** “이 문자들도 인식 대상에 넣어라”를 줄 수 있음.  
  - 형식: 한 줄에 한 글자씩 (UTF-8).  
  - 용도:  
    - 도메인 특화 글자·약어 (의회, 조례, 법률 용어 등)  
    - 기본 한글 사전에 없는 기호·특수문자  
  - **적용 방법:**  
    1. `ppocr/utils/ppocr_keys_v1.txt` 같은 기본 사전을 복사해,  
    2. 필요한 글자만 추가한 파일을 만들고,  
    3. `get_ocr_engine(rec_char_dict_path="경로/커스텀_사전.txt")` 로 넘기면 됨. (코드에서 이미 인자 지원함.)

즉, **“쉽게 성능 높이는 방법 중에 테서랙트 trained data처럼 쓰기 좋은 건 `rec_char_dict_path` + (필요 시) `drop_score` 조정**이다.

### 2.3 그밖에 검토해 볼 만한 것

- **업로드 파일 크기 제한**  
  매우 큰 PDF를 그대로 받으면 메모리/디스크 부담이 커짐.  
  FastAPI `File()` 에서 `max_length` 또는 미들웨어/의존성으로 최대 크기 제한을 두는 편이 안전함.

- **디지털 PDF 직접 추출**  
  텍스트 레이어가 있는 PDF는 OCR 없이 텍스트만 뽑는 경로를 두면 속도·품질 모두 유리함.  
  (별도 이슈로 “다이렉트 추출” 추가 예정이라고 하신 부분에 해당.)

- **이미지 전처리**  
  스캔 품질이 나쁠 때만 의미 있음.  
  - 대비·이진화 등은 “쉽게” 붙이기보다는, 실제로 품질 문제가 있을 때 단계적으로 도입하는 게 좋음.

## 3. 쉽게 적용할 수 있는 개선 (요약)

| 방법 | 난이도 | 효과 | 비고 |
|------|--------|------|------|
| **`show_log=False`** | 쉬움 | 로그 정리 | 선택 |
| **`use_gpu=True` 명시** | 쉬움 | 의도 명확화 | 선택 |
| **`drop_score` 조정** | 쉬움 | 오인식/누락 균형 | 문서 샘플로 0.4~0.7 구간 실험 권장 |
| **`rec_char_dict_path`** | 쉬움 | 도메인 글자·용어 인식 개선 | 테서랙트 tessdata처럼 “파일 하나 추가” 수준 |
| **업로드 크기 제한** | 쉬움 | 안정성 | 예: 50MB 등 |
| **디지털 PDF 직접 추출** | 중간 | 속도·품질 | 별도 구현 |
| **전처리(대비/이진화)** | 중간 | 저품질 스캔 | 필요 시만 |
| **모델 파인튜닝** | 어려움 | 특정 폰트/레이아웃 | 5000+ 레이블 샘플 등 필요 |
